# Optimize non-pack downloads and updater architecture

## Goal

Make downloads outside the user-managed integration-pack resources faster and more reliable, using PCL's proven download-engine ideas, then separate the updater's download concerns into maintainable modules.

## Confirmed Facts

- Integration-pack files are intentionally excluded from single-file Range splitting and must retain their existing single-connection/source-switch behavior.
- Minecraft client, libraries, native libraries, asset indexes, and asset objects are the scope for the new download engine.
- The only hard-coded Minecraft download source is the BMCLAPI main endpoint (`https://bmclapi2.bangbang93.com`). No Mojang, Forge, NeoForge, or hard-coded mirror fallback may be used.
- BMCLAPI URL rewriting is path-sensitive:
  - metadata (`piston-meta`/`launchermeta`) keeps its complete path and only changes host;
  - Maven artifacts use `/maven/<path>`;
  - asset objects use `/assets/<prefix>/<sha1>`.
- The prior asset-index bug came from inventing `/assets/indexes/<id>.json`; the correct path comes from `assetIndex.url`, with only the metadata host replaced.
- PCL's relevant ideas are a global request budget, source/request failure tracking, retries with backoff, resumable single-stream transfers, and bounded parallel local-file checks. Project policy forbids parallel file splitting for every source.

## Requirements

1. Build one shared non-pack download engine instead of maintaining separate generic and pack implementations.
2. Keep integration-pack downloads on the existing non-split path while allowing the shared engine to expose an explicit policy that disables splitting for them.
3. Centralize BMCLAPI URL construction and validate metadata-path rewrites so asset indexes cannot regress to filename-based concatenation.
4. Add robust HTTP handling for redirects, 403/404/408/429/5xx, truncated responses, Range mismatch, hash mismatch, cancellation/source-generation changes, and retry exhaustion.
5. Use a bounded global request budget and a separate file scheduler; avoid burst patterns that make the last pending library fail near completion.
6. Keep every file on one transfer stream. Range requests may only resume an existing single-stream temporary file, with size/hash validation and safe restart when the server rejects the resume request.
7. Preserve progress/speed events and existing Tauri command contracts.
8. Add unit tests for URL rewriting, Range partitioning, retry classification, and future/temporary-file safety; add an integration-style local HTTP test for Range success and fallback.

## Acceptance Criteria

- [ ] A full resource sync uses only BMCLAPI main URLs generated by the centralized source layer.
- [ ] Asset-index URLs generated from representative `piston-meta` paths return the expected BMCLAPI path shape and do not include `/assets` or `/indexes` incorrectly.
- [ ] Integration-pack downloads never enter the Range splitter.
- [ ] A transient 429/5xx or truncated response retries with bounded backoff and resumes a valid temporary file without creating parallel `.part` downloads.
- [ ] A failed final library does not abort solely because the scheduler exhausted a burst; the error identifies URL, status/classification, attempt count, and destination.
- [ ] Existing files with correct SHA-1 are skipped; incorrect files are replaced atomically after validation.
- [ ] `cargo check`, `cargo test`, and the frontend build pass.
- [ ] The updater module boundaries make source URL construction, scheduling, transfer, validation, and progress reporting independently testable.

## Resolved Decisions

- The architecture refactor may split `src-tauri/src/updater.rs` into dedicated Rust modules. This is approved because the user explicitly requested overall architecture optimization and asked to absorb PCL's download-engine practices.
